## 1. 目标与约束

本文件用于约定前端如何消费当前后端接口，并对齐已经落地的页面、鉴权、流式协议和 PWA 边界。

固定约束：

- 前端目录：`D:\Git\agent\noval\frontend`
- Node 版本：`20 LTS`
- 鉴权方式：`Authorization: Bearer <token>`
- 当前业务平台：`fanqie`
- 后端为单 token 模式，不设计 `refreshToken`
- 用户角色来源于 JWT claims，不额外请求“当前用户”接口

## 2. 统一响应与错误处理

```ts
export interface ApiResponse<T> {
  code: number;
  message: string;
  data: T;
  timestamp: number;
  traceId: string;
}
```

前端处理规则：

- 优先看 HTTP Status
- `code` 作为 JSON 兜底校验
- 页面错误提示保留 `traceId`
- `401` 触发 refresh，一次失败后清理会话并跳转 `/login`

## 3. JWT 与会话设计

后端当前真实返回：

```ts
export interface TokenResponse {
  accessToken: string;
  tokenType: 'Bearer';
  expiresIn: number;
}
```

本地存储键：

```ts
export const ACCESS_TOKEN_KEY = 'noval.access_token';
export const TOKEN_TYPE_KEY = 'noval.token_type';
export const TOKEN_EXPIRE_AT_KEY = 'noval.token_expire_at';
```

登录成功后，前端解码 JWT payload，构造本地会话：

```ts
export interface AuthSession {
  userId: number;
  username: string;
  roles: Array<'ADMIN' | 'USER'>;
  accessToken: string;
  tokenType: 'Bearer';
  expiresIn: number;
  expireAt: number;
}
```

刷新策略：

- 普通请求：`401 -> refresh -> retry once`
- 流式请求：开流前若 60 秒内过期则先 refresh
- 流式收到 `401` 时仅允许 refresh 并重开一次

## 6. API 模块设计

### 6.1 Auth

- `authApi.login`
- `authApi.refresh`
- `authApi.logout`

接口：

- `POST /api/auth/login`
- `POST /api/auth/refresh`
- `POST /api/auth/logout`

说明：

- `refresh` 只传 body `{ token }`
- `logout` 同时传 header token 和 body token

### 6.2 Crawler

- `crawlerApi.getBookDetail`
- `crawlerApi.getChapters`

- `POST /api/crawler/chapters`

### 6.3 Analysis

- `analysisApi.runDeconstruct`

- `analysisApi.runPlot`
- `analysisApi.streamDeconstruct`
- `analysisApi.streamStructure`
- `analysisApi.streamPlot`

- `analysisApi.streamTrend`

接口：

- `POST /api/analysis/deconstruct`
- `POST /api/analysis/structure`
- `POST /api/analysis/plot`

- `POST /api/analysis/structure/stream`
- `POST /api/analysis/plot/stream`
- `GET /api/analysis/trend`

### 6.4 Data

- `dataApi.getHistory`
- `dataApi.getVisual`

- `GET /api/data/history`
- `GET /api/data/visual`

### 6.5 Config

- `configApi.getPromptConfig`
- `configApi.updatePromptConfig`
- `configApi.getSystemConfig`
- `configApi.updateSystemConfig`

接口：

- `GET /api/config/prompt?promptType=deconstruct`
- `PUT /api/config/prompt`
- `GET /api/config/system?configKey=ai.timeout.millis`
- `PUT /api/config/system`

说明：

- `PUT /api/config/prompt` 本轮新增可读可写字段：`temperature`、`maxTokens`
- `modelName` 直接表示模型名，例如 `deepseek-chat`
- AI 供应商级非密钥配置通过 `system_config` 固定 key 管理
- 模型密钥仍只放本地环境变量，不通过接口下发或回显

## 7. SSE 协议

前端统一通过 `fetch + ReadableStream` 解析 SSE。

事件格式：

```ts
export type StreamEventType = 'start' | 'delta' | 'done' | 'error';

export interface StreamStartEvent {
  event: 'start';
  traceId: string;
  analysisType: 'deconstruct' | 'structure' | 'plot' | 'theme';
}

export interface StreamDeltaEvent {
  event: 'delta';
  delta: string;
  chunkIndex?: number;
}

export interface StreamDoneEvent<T> {
  event: 'done';
  data: T;
}

export interface StreamErrorEvent {
  event: 'error';
  code: number;
  message: string;
  traceId?: string;
}
```

当前已实现流式接口：

- `POST /api/analysis/deconstruct/stream`
- `POST /api/analysis/structure/stream`
- `POST /api/analysis/plot/stream`
- `POST /api/analysis/trend/stream`

补充说明：

- 本轮后端已接入 `LangChain4j + OpenAI 兼容模型`
- SSE 接口路径不变，前端无需改流式路由
- 当前仍按协议流消费，不要求前端切换到厂商私有流式协议

## 8. 流式回退策略

前端固定策略：

1. 优先请求流式接口
2. 若返回 `404 / 405 / 501`
3. 或 `Content-Type` 不是 `text/event-stream`
4. 或在收到任何 `delta` 前解析失败
5. 则自动回退到阻塞接口

流中断规则：

- 若已收到部分 `delta`，保留已生成内容
- 显示“连接中断，可重试”
- 不自动切回阻塞接口，避免重复内容

## 9. 页面行为约定

### 9.1 `/analysis`

- 从路由 query 读取 `bookId / chapterCount / platform / mode`
- 三种模式共用同一结果卡与工具栏
- 结果卡支持：
  - 流式文本
  - 闪烁光标
  - 停止生成
  - 重新生成
  - 复制结果

### 9.2 `/trend`

- 默认 `platform = fanqie`
- 默认 `category = male-hot-a`
- 页面初始化并行请求：
  - `GET /api/data/visual?platform=fanqie`
  - `POST /api/analysis/trend/stream`
- 分类切换时重新发起趋势流式分析
- 页面包含：
  - 趋势结果卡
  - 趋势摘要卡
  - 图表区
  - 最新快照表
  - 标签云
  - 主题对比列表

### 9.3 `/history`

- 默认请求：`GET /api/data/history?platform=fanqie&limit=20`
- 支持 `analysisType / bookId / limit`
- 详情直接消费列表项，不重新生成分析
- 移动端详情使用底部抽屉

## 10. PWA 设计边界

当前已实现：

- `vite-plugin-pwa`
- manifest
- Service Worker 注册
- 生产构建输出 `sw.js`

缓存边界：

允许缓存：

- 静态壳层资源
- `GET /api/data/visual`

默认不缓存：

- `POST /api/analysis/*/stream`
- `GET /api/analysis/trend`
- `GET /api/data/history` 的完整正文
- 完整 AI 长文本结果

原因：

- 趋势与历史结果属于受保护的内部分析数据
- 不做默认离线持久化，降低泄露风险
