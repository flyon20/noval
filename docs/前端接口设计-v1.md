## 2. 公共协议

### 2.1 基础约束

- 基础路径：`/api`
- 认证头：`Authorization: Bearer <token>`
- 默认平台：`fanqie`
- 前端不使用 Cookie 会话，不依赖服务端 Session。

### 2.2 统一响应结构

```ts
export interface ApiResponse<T> {
  code: number;
  message: string;
  data: T;
  timestamp: number;
  traceId: string;
}
```

后端当前统一返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {},
  "timestamp": 1760000000000,
  "traceId": "f0b5c0aa2cbe4f58a7d0f8f012345678"
}
```

### 2.3 HTTP 状态码与前端处理

| HTTP | 场景 | 前端动作 |
| --- | --- | --- |
| `200` | 成功 | 正常渲染 |
| `400` | 参数错误、表单校验失败 | 表单提示或页内错误提示 |
| `401` | 未登录、Token 无效、Token 过期 | 先尝试 refresh，一次失败后清空登录态并跳转 `/login` |
| `403` | 无权限、只读配置不可修改 | 展示无权限态 |
| `404` | 资源不存在、接口未实现 | 页面空态或流式兜底 |
| `429` | 限流触发 | 顶部消息 + 禁用重复提交 |
| `500` | 服务端异常 | 展示通用错误并保留 `traceId` |

规则：

1. 前端错误处理优先使用 HTTP Status。
2. `response.code` 作为 JSON 兜底校验。
3. 所有错误弹窗或错误页都要展示 `traceId`，便于后端排查。

## 3. JWT、鉴权与会话恢复

### 3.1 后端当前真实行为

后端当前为单 token 模式，登录和刷新都只返回：

```ts
export interface TokenResponse {
  accessToken: string;
  tokenType: 'Bearer';
  expiresIn: number;
}
```

后端没有 `refreshToken`，也没有“当前用户信息”接口。

### 3.2 LocalStorage 键名

前端固定使用以下键名：

```ts
export const ACCESS_TOKEN_KEY = 'noval.access_token';
export const TOKEN_TYPE_KEY = 'noval.token_type';
export const TOKEN_EXPIRE_AT_KEY = 'noval.token_expire_at';
```

### 3.3 JWT Payload 解码

前端登录成功后必须本地解码 JWT payload，构造会话对象。

```ts
export interface JwtClaims {
  sub: string;
  uid: number;
  username: string;
  roles: string;
  iat: number;
  exp: number;
}

export type RoleCode = 'ADMIN' | 'USER';

export interface AuthSession {
  userId: number;
  username: string;
  roles: RoleCode[];
  accessToken: string;
  tokenType: 'Bearer';
  expiresIn: number;
  expireAt: number;
}
```

处理规则：

1. `roles` 是逗号分隔字符串，前端要 `split(',')` 后过滤空值。
2. `expireAt` 由当前时间加 `expiresIn` 秒计算，同时保留 JWT 自身 `exp` 作为调试参考。
3. 刷新页面时从 LocalStorage 恢复 token，再重新解码构造 `AuthSession`。

### 3.4 请求拦截器

所有受保护接口统一：

1. 从 LocalStorage 读取 token。
2. 自动追加 `Authorization: Bearer <token>`。
3. 未登录时直接阻止进入核心业务页。

### 3.5 Refresh 规则

#### 普通请求

1. 请求先正常发送。
2. 若返回 `401`，且当前请求不是 `/api/auth/login`、`/api/auth/refresh`：
   - 使用当前 token 调用 `/api/auth/refresh`
   - refresh 成功后覆盖本地 token
   - 原请求仅自动重放一次
3. 若 refresh 失败，则清空登录态并跳转 `/login`。

#### 流式请求

1. 开流前如果 `expireAt - now <= 60_000ms`，先主动 refresh。
2. 流式请求收到 `401` 时，只允许 refresh 并重开一次。
3. 如果 refresh 仍失败，则终止流式输出并退回登录页。

### 3.6 登录、刷新、登出接口注意事项

#### 登录

```ts
export interface LoginRequest {
  username: string;
  password: string;
}
```

- `POST /api/auth/login`
- 无需 `Authorization`
- 可能返回 `429`，前端需提示“登录过于频繁”

#### 刷新

```ts
export interface RefreshTokenRequest {
  token: string;
}
```

- `POST /api/auth/refresh`
- 使用请求体里的 `token`
- 不依赖 `Authorization` 头

#### 登出

- `POST /api/auth/logout`
- 必须同时带：
  - `Authorization: Bearer <currentToken>`
  - 请求体 `{ token: currentToken }`
- 成功后前端立即清空本地会话

## 8. 已落地的协议流分析接口

### 8.1 目标

前端目标是实现主流 AI 产品的“边生成边显示”体验，但必须兼容当前后端仅支持阻塞接口的现状。

策略固定为：

1. 优先尝试流式接口。
2. 若流式接口未实现或网关不支持，则回退到阻塞接口。
3. 历史回看页不触发重新生成，不使用流式。

### 8.2 已落地接口

| 接口 | 方法 | 状态 | 请求格式 | 说明 |
| --- | --- | --- | --- | --- |
| `/api/analysis/deconstruct/stream` | `POST` | 已落地（协议流） | `AnalysisRequest` | 流式拆文 |
| `/api/analysis/structure/stream` | `POST` | 已落地（协议流） | `AnalysisRequest` | 流式拆结构 |
| `/api/analysis/plot/stream` | `POST` | 已落地（协议流） | `AnalysisRequest` | 流式拆情节 |
| `/api/analysis/trend/stream` | `POST` | 已落地（协议流） | `{ platform: 'fanqie'; category?: string }` | 流式题材趋势 |

流式响应固定：

- `Content-Type: text/event-stream;charset=UTF-8`
- 前端通过 `fetch + ReadableStream` 解析，不使用 `EventSource`

### 8.3 SSE 事件类型

```ts
export type StreamEventType = 'start' | 'delta' | 'done' | 'error';

export interface StreamStartEvent {
  event: 'start';
  traceId: string;

}

  event: 'delta';

  chunkIndex?: number;
}

  event: 'done';

}

export interface StreamErrorEvent {
  event: 'error';
  code: number;
  message: string;

```

SSE 原始示例：

```text

event: delta

event: done

```

### 8.5 流式失败与兜底规则

前端固定按以下顺序处理：

1. 开流前检查 token，若 60 秒内过期则先 refresh。
2. 发起流式请求。
3. 若初始响应返回 `404/405/501`，直接改走阻塞接口。
4. 若初始响应 `Content-Type` 不是 `text/event-stream`，直接改走阻塞接口。
5. 若流式解析在收到任何 `start/delta` 之前失败，直接改走阻塞接口。
6. 若已收到 `delta` 后网络中断：
   - 保留已生成文本
   - 页面状态改为“连接中断，可重试”
   - 不自动切到阻塞接口，避免重复内容
7. 若流式请求返回 `401`：
   - 只允许 refresh 并重开一次
   - 第二次仍失败则清空登录态并跳转 `/login`
8. 历史页不触发任何流式重放。

### 8.6 协议流与阻塞兜底说明

当前后端已提供 SSE 协议流接口，但仍不是模型侧 token 级真流式：

1. 后端先发送 `start` 事件。
2. 后端复用现有阻塞分析与落库逻辑，拿到完整结果后切分为多段 `delta` 事件。
3. 后端最后发送 `done` 事件，最终展示仍以 `done.data` 中的完整 `AnalysisResult` / `TrendAnalysisResult` 为准。
4. 若流式接口不可用、网关不支持或前置阶段失败，前端仍按阻塞接口兜底。

### 9.3 分析页 `/analysis`

目标：

1. 针对单本书执行 `deconstruct / structure / plot`。
2. 优先使用流式协议。
3. 支持停止生成、重新生成、复制结果。

页面状态：

```ts
type AnalysisStreamPhase = 'idle' | 'preparing' | 'streaming' | 'fallback-blocking' | 'done' | 'error';

interface AnalysisPageState {
  bookId: number;
  platform: Platform;
  chapterCount: UiChapterCount;
  activeAnalysisType: 'deconstruct' | 'structure' | 'plot';
  phase: AnalysisStreamPhase;
  streamingText: string;
  result?: AnalysisResult;
  errorMessage?: string;
  traceId?: string;
}
```

视觉约定：

1. 结果卡片使用 markdown 渲染。
2. 流式输出时显示闪烁光标。
3. 增量渲染采用节流更新，避免每个字符都重排。
4. 完成后显示 `traceId`、模型名、`tokenUsed`。

### 9.4 历史页 `/history`

目标：

1. 查看分析历史。
2. 按 `bookId / analysisType / limit` 过滤。
3. 点击列表项直接展开详情，不二次请求。

页面状态：

```ts
interface HistoryPageState {
  filters: {
    platform?: Platform;
    bookId?: number;

    limit: number;
  };
  items: AnalysisHistoryItem[];
  activeItem?: AnalysisHistoryItem;
}
```

说明：

1. 历史页只读。
2. 历史结果展示完整 `resultContent`，`resultJson` 作为结构化面板。
3. 不提供“重新生成”。

### 9.5 趋势页 `/trend`

目标：

1. 加载可视化数据。
2. 生成题材趋势分析。
3. 支持流式优先与阻塞兜底。

页面默认：

1. `platform = 'fanqie'`
2. `category = 'all'`

页面状态：

```ts
interface TrendPageState {
  platform: Platform;
  category: TrendCategoryFilter;
  visual?: VisualData;
  phase: AnalysisStreamPhase;
  streamingText: string;
  trendResult?: TrendAnalysisResult;
  errorMessage?: string;
  traceId?: string;
}
```

接口映射：

1. 图表区域使用 `/api/data/visual?platform=fanqie`
2. 趋势分析区域：
   - 流式优先：`POST /api/analysis/trend/stream`
   - 阻塞兜底：`GET /api/analysis/trend`
3. 当 UI 选择“全部”时，不传 `category`

### 9.6 提示词配置页 `/config/prompt`

目标：

1. 按类型读取提示词。
2. 单项编辑并保存。

页面约束：

1. 固定标签页：`deconstruct / structure / plot / theme`
2. 无列表接口，前端必须按固定类型逐个读取
3. 提交前校验 `{{content}}`

### 9.7 系统配置页 `/config/system`

目标：

1. 管理固定系统配置项。
2. 只向 `ADMIN` 展示。

页面约束：

1. 无列表接口，前端固定展示 3 个已知 key。
2. 若返回 `editable=false`，则前端把表单置为只读。

## 10. 前端实现前检查清单

1. 新建前端工程时目录必须是 `D:\Git\agent\noval\frontend`。
2. Node 版本通过 `nvm` 管理，默认 Node 20 LTS。
3. 所有受保护接口都走统一请求封装。
4. `logout` 必须同时发 header token 与 body token。
5. `refresh` 只提交 body token，不设计 `refreshToken`。
6. `system config` 页面不得对普通用户暴露入口。
7. 流式章节必须明确标记为“协议流已上线，但非模型真流式”，不可误写为模型直推实时流。
