# 榜单共享快照与自动分析设计说明

## 目标

在保持当前项目仍以 `fanqie` 为唯一平台的前提下，重做“扫榜 -> 入库 -> 分页展示 -> 榜单级 AI 分析 -> 多用户复用”这条链路，满足以下目标：

- 榜单抓取从“单个占位 category”升级为“频道 + 榜单”的真实目录模型。
- 每次抓取一个榜单时，一次抓完整个榜单并入库，前端分页只查数据库，不重复触发爬虫。
- 榜单数据在多用户之间共享，避免每个用户重复抓取同一个榜单。
- 榜单抓取完成后自动触发榜单级 AI 分析，统计、词云、题材表和摘要与当前榜单快照保持一致。
- 榜单分析结果有独立缓存期；切换榜单时，若当前榜单最近一次分析仍在缓存期内，则直接展示数据库结果，过期后再基于最新快照重算。
- 用户对当前榜单分析结果不满意时，可手动触发重新分析，但次数受管理员配置限制，默认按 3 次理解。
- 用户登录后按“启用榜单清单”自动检查是否需要补抓；若缓存期内已有其他用户抓取的快照，则直接复用。
- 登录页精简为最小必要信息，不再展示与登录动作无关的流程性说明。
- 关键决策、频控命中、抓取入库、自动分析、分析缓存命中和失败原因都要写日志，便于追踪。

## 当前问题

- 当前前端扫榜页把榜单分类写死为 `male-hot-a`、`male-hot-b`、`male-new-a` 三个占位值，无法表达真实的“频道 + 榜单”结构。
- `重新扫榜` 前端没有传强刷语义，后端会因为 `AUTO` 复用策略和历史快照/缓存而表现为“点了没反应”。
- 榜单列表当前直接绑定最新快照结果，没有快照分页概念，后续翻页只能再次抓取，违背业务预期。
- Python crawler 只内置少量占位 alias，不支持真实榜单目录发现和具体榜单编码清洗。
- 榜单题材分析当前仅基于最近三次快照拼文本，`/api/data/visual` 又优先读取全平台最近一次 `theme` 结果，无法保证和当前选中榜单一致。
- 当前系统没有“分析缓存是否过期”的判断，也没有“手动重分析次数限制”的控制，难以兼顾结果复用与成本控制。
- 当前测试初始化数据中写入了固定榜单和固定题材结果，容易让页面长期显示重复样本，掩盖真实抓取链路问题。
- 登录页目前包含过多介绍性文案，影响聚焦登录动作本身。

## 设计边界

- 本轮同时覆盖前端、后端、crawler、SQL 和测试。
- 仍然只支持 `fanqie`，不扩展第二个平台。
- 暂不引入新的异步任务中间件或外部队列，自动抓取和自动分析先复用现有应用内执行模型。
- 书级详情、章节、单书分析继续存在，但公共榜单数据与书级缓存要显式分层。
- 不要求本轮做复杂调度中心；登录自动补抓和手动抓榜优先落地。

## 核心方案

### 1. 榜单目录模型

榜单维度从一个 `category` 字符串升级为两层：

- `channelCode` / `channelName`
- `boardCode` / `boardName`

其中：

- `channel` 表示频道，例如男频新书榜、男频阅读榜。
- `board` 表示某个频道下的具体榜单，例如都市脑洞、玄幻脑洞。
- `boardCode` 在同一平台下唯一标识一个具体榜单，用于抓取、快照、分析和分页查询。

Python crawler 在抓取榜单数据时，需要同时支持：

- 榜单目录发现与清洗
- 指定 `channelCode + boardCode` 抓取完整榜单

后端和前端均不再写死榜单枚举。

### 2. 一次抓完整榜，前端只查库分页

抓取逻辑调整为：

1. 用户或系统触发某个榜单刷新。
2. 后端调用 crawler 一次抓完整个榜单的全部书目。
3. 后端创建一条榜单快照记录。
4. 后端把该次抓取的完整榜单条目批量写入数据库，并挂到该快照下。
5. 前端后续翻页仅针对该快照做数据库分页查询，不再触发爬虫。

这意味着：

- “翻到下一页”永远是数据库分页行为。
- “重新抓榜”永远是完整榜单刷新行为。
- 当前展示的是某个明确 `snapshotId` 对应的数据。

### 3. 多用户共享公共榜单数据

榜单、书籍、章节缓存都按公共数据处理，不按用户隔离。

共享规则：

- 如果榜单在缓存有效期内已有最新快照，则所有用户都复用该快照。
- 如果某本书的详情或章节在缓存有效期内已经抓过，则后续用户直接复用。
- 只有用户主动触发重新抓取或重新分析，且满足频控条件时，才重新访问 crawler 或 AI。

用户差异仅体现在：

- 登录会触发哪些启用榜单的自动检查。
- 用户自己的手动操作行为记录。
- 某些界面上的最近查看状态。

公共数据不应因为用户数量增加而被重复抓取。

### 4. 登录后的自动补抓

系统配置中维护一组“启用榜单清单”。

用户登录成功后：

1. 后端读取启用榜单清单。
2. 对每个榜单检查最新快照时间。
3. 若快照仍在 `rank.login-auto-refresh-days` 范围内，则直接复用。
4. 若快照过期或不存在，则尝试自动抓取。
5. 自动抓取成功后立即触发该榜单的自动分析。

这样可以保证：

- 长时间未登录后重新进入系统，页面看到的是相对新的榜单。
- 若其他用户刚刚更新过同一榜单，则当前用户不会重复触发抓取。
- 自动补抓范围可控，不会在登录时无脑抓取所有榜单。

### 5. 榜单抓取频控

频控按“平台 + 榜单”维度执行，而不是全平台共用一个冷却窗口。

核心规则：

- 同一榜单在有效缓存期内，`AUTO` 模式直接复用最新快照。
- 同一榜单在冷却窗口内，不允许无上限 `FORCE` 重抓。
- 切换到另一个榜单时，应按另一个榜单自己的频控和缓存独立判断，不互相影响。

后端返回时需要显式告诉前端：

- 当前是否复用了已有快照。
- 当前是否命中频控限制。
- 限制原因是什么。
- 当前页面正在展示的快照是谁。

### 6. 榜单抓取完成后自动分析

抓榜完成后自动触发一次榜单级 AI 分析。

分析输入是当前榜单快照的完整条目集合，至少包括：

- 频道名
- 榜单名
- 快照时间
- 每本书的排名
- 书名
- 作者
- 简介

分析输出不再只是一段自由文本，而是要求模型输出符合约定 JSON 的结构化结果，并绑定到当前 `snapshotId`。

前端显示的榜单统计、词云、题材表、榜单摘要必须与当前展示快照一致，不能继续读取“全平台最近一次 theme 分析”作为兜底主数据。

### 7. 榜单分析缓存与手动重分析

榜单分析缓存与榜单抓取缓存分开管理。

规则如下：

- 切换到某个榜单时，先取该榜单最新快照绑定的最近一次分析结果。
- 若该分析结果仍在 `analysis.rank.reuse-days` 定义的缓存期内，则直接展示数据库结果。
- 若分析结果不存在，或已超过缓存期，则基于当前最新快照重新分析。
- 若用户对当前结果不满意，可点击“重新分析”手动触发重跑。
- 手动重分析只针对当前榜单当前快照，不重新抓榜。
- 手动重分析次数受管理员配置限制，默认按 3 次理解，应通过系统配置调整。

这样可以保证：

- 切换榜单时优先复用已有结果，避免不必要的大模型调用。
- 当榜单快照已更新或分析缓存已过期时，能自动对齐到最新数据。
- 用户仍保留有限次人工纠偏能力。

## 数据模型

### 1. `rank_board`

新增榜单目录表，用于维护真实榜单目录。

建议字段：

- `id`
- `platform`
- `channel_code`
- `channel_name`
- `board_code`
- `board_name`
- `source_path`
- `enabled`
- `sort_order`
- `last_sync_time`
- `create_time`
- `update_time`
- `deleted`

职责：

- 提供前端频道/榜单选择源数据。
- 提供登录自动补抓的启用清单。
- 提供 crawler 目录同步落库能力。

### 2. `rank_snapshot`

新增榜单快照头表，用于记录每次完整抓榜结果。

建议字段：

- `id`
- `platform`
- `channel_code`
- `channel_name`
- `board_code`
- `board_name`
- `snapshot_time`
- `book_count`
- `trigger_type`
- `status`
- `reused_from_snapshot_id`
- `source_hash`
- `trace_id`
- `error_message`
- `create_time`
- `update_time`
- `deleted`

职责：

- 为分页查询提供稳定的快照维度。
- 为榜单分析提供明确锚点。
- 记录抓取来源、复用情况和异常。

### 3. `crawl_rank`

保留榜单条目表，但调整为强绑定快照。

新增/调整字段：

- `snapshot_id`
- `platform`
- `channel_code`
- `board_code`
- `rank_no`
- `book_id`
- `book_name`
- `author`
- `intro`
- `book_url`
- `crawl_time`

职责：

- 存储某个快照下的完整榜单条目。
- 为前端分页提供基础数据。

### 4. `crawl_book` 与 `crawl_chapter`

继续作为公共缓存表，不按用户隔离。

规则：

- 榜单抓取时同步更新 `crawl_book` 基础信息。
- 点进书详情时优先复用 `crawl_book`。
- 抓章时优先复用 `crawl_chapter`。
- 手动重抓和缓存过期时才重新访问 crawler。

### 5. `analysis_result`

建议扩展为同时支持书级分析和榜单级分析。

新增字段：

- `scope_type`，例如 `BOOK` / `RANK_BOARD`
- `snapshot_id`
- `channel_code`
- `board_code`
- `trigger_type`
- `trigger_user_id`
- `reanalysis_seq`

榜单分析记录需要和具体 `snapshotId` 绑定，保证展示数据来源明确，也便于统计手动重分析次数。

## 接口设计

### 1. 榜单目录

`GET /api/crawler/boards?platform=fanqie`

返回：

- 可选频道列表。
- 每个频道下的榜单列表。
- 每个榜单的启用状态。
- 最近快照时间与新鲜度标记。

用于前端渲染真实目录，不再依赖硬编码。

### 2. 榜单刷新

`POST /api/crawler/rank/refresh`

请求：

- `platform`
- `channelCode`
- `boardCode`
- `refreshMode`
- `triggerType`

响应建议包含：

- `snapshotId`
- `snapshotTime`
- `bookCount`
- `reused`
- `refreshLimited`
- `refreshLimitedReason`
- `analysisTriggered`

语义：

- 若命中缓存并允许复用，则直接返回已有快照。
- 若需要抓取，则一次抓完整榜单、入快照、自动分析。

### 3. 榜单分页

`GET /api/crawler/rank/page`

请求：

- `platform`
- `channelCode`
- `boardCode`
- `snapshotId` 可选
- `page`
- `pageSize`

响应：

- `snapshotId`
- `snapshotTime`
- `total`
- `page`
- `pageSize`
- `items`

语义：

- 默认查询该榜单最新可用快照。
- 只查数据库，不触发爬虫。

### 4. 榜单分析

`GET /api/analysis/rank/latest`

请求：

- `platform`
- `channelCode`
- `boardCode`
- `snapshotId` 可选

返回：

- `analysisStatus`
- `snapshotId`
- `modelName`
- `resultContent`
- `resultJson`
- `cached`
- `cacheExpired`
- `manualReanalyzeRemaining`

语义：

- 若当前快照最近一次分析仍在缓存期内，则直接返回数据库结果。
- 若分析已过期，则由后端基于最新快照触发重算，并返回当前状态。

`POST /api/analysis/rank/reanalyze`

请求：

- `platform`
- `channelCode`
- `boardCode`
- `snapshotId`
- `forceReanalyze`

语义：

- 只基于当前 `snapshotId` 重跑，不重新抓榜。
- 若管理员配置的手动重分析次数已耗尽，则返回受限提示。

### 5. 榜单可视化

`GET /api/data/visual/rank`

请求：

- `platform`
- `channelCode`
- `boardCode`
- `snapshotId` 可选

返回：

- `summary`
- `wordCloud`
- `themeTable`
- `snapshotComparisons`
- `rankCategoryDistribution`
- 其他榜单相关统计

数据必须来自当前榜单快照绑定的分析结果，而不是平台级最近一条 `theme` 记录。

### 6. 登录后的自动检查

`POST /api/system/login-bootstrap`

行为：

- 检查启用榜单清单。
- 对每个榜单决定复用、自动抓取或跳过。
- 返回处理结果列表供前端提示。

## 自动抓取与自动分析时序

### 1. 登录时

1. 登录成功。
2. 后端读取启用榜单清单。
3. 检查每个榜单的最新快照是否过期。
4. 若未过期则复用。
5. 若过期则自动抓取完整榜单。
6. 抓取成功后自动触发榜单分析。

### 2. 进入扫榜页时

1. 查询榜单目录。
2. 默认选中一个启用榜单。
3. 查询该榜单最新快照分页数据。
4. 查询该榜单最新分析结果与可视化数据。

分析查询规则：

- 若当前榜单最新分析仍在缓存期内，则直接返回数据库结果。
- 若分析已过期，则基于当前最新快照自动重算。

### 3. 手动重抓时

1. 前端发起 `rank/refresh`。
2. 后端按榜单维度判断缓存和频控。
3. 允许抓取时执行完整抓榜。
4. 入库快照和榜单条目。
5. 自动触发分析。
6. 前端刷新快照分页和分析状态。

### 4. 翻页时

1. 前端只发分页查询。
2. 后端按 `snapshotId` 查库。
3. 不触发抓取。

### 5. 手动重新分析时

1. 前端对当前榜单当前快照发起 `rank/reanalyze`。
2. 后端先检查该快照的手动重分析次数是否已达上限。
3. 若未达上限，则直接复用当前快照数据重跑 AI。
4. 保存新的榜单分析记录，并更新展示结果。
5. 若已达上限，则返回明确受限原因与剩余次数信息。

## Prompt 与 JSON 契约

榜单级分析必须要求模型输出稳定 JSON，而不是完全自由文本。

建议在 `prompt_config` 中扩展：

- `output_schema`
- `output_example`
- `post_process_type`

榜单分析最少要求返回：

- `summary`
- `wordCloud`
- `themeTable`
- `snapshotComparisons`
- `insights`
- `riskNotes`

AI 网关处理顺序：

1. 读取 Prompt 模板。
2. 拼入榜单快照内容。
3. 拼入输出 schema 和 example。
4. 调用模型。
5. 提取 JSON。
6. 校验结构。
7. 填充默认值。
8. 存入 `analysis_result.result_json`。

若模型未完全遵守 schema：

- 后端做兜底清洗。
- 标记日志。
- 允许保存部分可用结果。
- 避免整条链路因为格式波动完全失败。

榜单分析缓存判定基于“同一 `snapshotId` 的最新有效分析结果”进行，不同快照之间不共享分析缓存。

## 系统配置建议

建议新增或明确以下配置项：

- `rank.snapshot.reuse-days`
- `rank.login-auto-refresh-days`
- `rank.force-cooldown-days`
- `rank.force-max-times`
- `rank.enabled.boards`
- `rank.default-page-size`
- `book.detail.reuse-days`
- `book.chapter.reuse-days`
- `analysis.rank.auto-enabled`
- `analysis.rank.reuse-days`
- `analysis.rank.manual-max-times`
- `analysis.book.reuse-hours`
- `analysis.book.manual-max-times`

其中：

- `analysis.rank.reuse-days` 控制榜单分析缓存期。
- `analysis.rank.manual-max-times` 控制同一榜单快照可手动重分析的次数上限。
- `rank.force-cooldown-days` 与 `rank.force-max-times` 控制同一榜单的手动抓取限制。
- `book.detail.reuse-days` 与 `book.chapter.reuse-days` 控制书详情、章节缓存期。
- `analysis.book.reuse-hours` 与 `analysis.book.manual-max-times` 预留给单书分析复用和手动重跑限制。

这些值都不应继续写死在代码常量中，而应由管理员通过系统配置维护。

## 前端改造

### 1. 登录页

保留：

- 项目名
- 一句功能说明
- 用户名
- 密码
- 错误提示

移除：

- 与登录动作无关的功能卖点卡片
- 过长的流程说明文案

### 2. 扫榜页

顶部区域：

- 频道选择
- 榜单选择
- 最近快照时间
- 重新抓取按钮
- 重新分析按钮
- 频控或缓存命中提示
- 分析状态提示
- 剩余手动重分析次数提示

列表区域：

- 当前快照分页结果
- 默认每页 10 本
- 展示总数与分页组件

分析区域：

- 当前榜单摘要
- 词云
- 题材表
- 最近几次快照对比
- 失败时允许重新分析

### 3. 单书链路

从榜单列表继续进入单书详情、抓章和单书分析。

但其数据规则独立于榜单分页：

- 榜单页只负责公共榜单数据。
- 书级缓存和单书分析按书维度复用。

### 4. 管理员配置入口

系统配置页需要覆盖与本链路相关的管理项，至少包括：

- 榜单自动复用缓存期
- 登录自动补抓阈值
- 榜单手动抓取冷却期与次数上限
- 书详情缓存期
- 章节缓存期
- 榜单分析缓存期
- 榜单手动重分析次数上限

这样管理员才能按实际成本、抓取频率和运营需求调整策略，而不是依赖改代码发版。

## 日志设计

日志必须能还原“为什么这次抓或不抓、分析或不分析”。

至少覆盖：

- 登录触发自动检查
- 榜单目录同步
- 榜单抓取决策
- 命中缓存复用
- 命中频控限制
- crawler 调用开始/成功/失败
- 快照写库成功/失败
- 自动分析开始/成功/失败
- 榜单分析缓存命中/过期判断
- 手动重分析次数校验与剩余次数
- 当前生效的关键配置值
- JSON 解析和 schema 校验失败

每条关键日志至少带：

- `traceId`
- `platform`
- `channelCode`
- `boardCode`
- `snapshotId`
- `triggerType`
- `refreshMode`
- `analysisTriggerType`
- `manualReanalyzeRemaining`
- `configVersion` 或关键阈值快照

禁止记录敏感信息和无必要的大段原始模型输入。

## 测试策略

### 后端单元测试

- 榜单缓存复用判定
- 榜单频控判定
- 多用户共享快照复用
- 登录自动补抓决策
- 榜单分析 JSON 校验与兜底
- 榜单分析缓存过期判定
- 榜单手动重分析次数限制

### 后端集成测试

- 抓完整榜单并写入快照
- 分页查询不触发 crawler
- 切换榜单后数据和分析跟随变化
- 登录后自动补抓能命中已有快照复用
- 榜单分析未过期时直接返回数据库结果
- 榜单分析过期后基于最新快照重算

### 前端测试

- 登录页精简渲染
- 频道与榜单联动
- 翻页只调用分页接口
- 强刷受限提示
- 当前榜单分析状态渲染
- 手动重新分析按钮与剩余次数展示

### 联调验证

- A 用户抓过榜单后，B 用户登录直接复用
- 同一榜单翻页不重新抓
- 切换新榜单时触发新快照与自动分析
- 抓榜完成后词云和题材表同步更新
- 切换到分析缓存未过期的榜单时直接展示库内结果
- 超过手动重分析上限后返回受限提示

## 实施注意事项

- 现有 `category` 相关 DTO、VO、前端类型和 crawler alias 都要迁移为 `channelCode + boardCode`。
- 现有测试种子中的占位榜单和固定 theme 结果会干扰真实联调，需要改成围绕快照和榜单分析的新数据模型。
- `GET /api/data/visual` 的平台级聚合逻辑不再适合作为扫榜页主数据源，建议拆出榜单维度接口。
- 自动补抓需要限制只处理启用榜单清单，避免登录时触发过多外部请求。
- 频控粒度必须落在具体榜单，不要误伤其他榜单切换。
- 榜单分析缓存期与手动重分析次数上限都应进入系统配置，由管理员维护。
- 榜单抓取次数限制、书级缓存期、分析缓存期等阈值都应在系统配置中可见、可改、可审计。
