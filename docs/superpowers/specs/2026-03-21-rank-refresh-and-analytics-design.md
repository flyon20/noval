# 榜单刷新与数据分析补齐设计

## 目标

在保持 V1 只支持 `fanqie` 的前提下，补齐当前后端尚未落地的业务能力：

- 榜单、书籍详情、章节内容不再只依赖 Redis TTL，而是以数据库快照为主数据源。
- 用户点击“扫榜”时遵循业务刷新策略，避免短时间重复抓取。
- 提供有限的强制刷新入口，用于平台提前更新或异常修链。
- 拆文类 AI 分析允许复用已抓取章节并在短期内重新分析。
- 数据分析接口返回词云、热门题材表、最近三次对比等结构化数据。

## 问题现状

- 现有榜单、详情、章节逻辑在缓存 miss 时会直接重新调用 Python 爬虫，没有“数据库快照可复用”的决策层。
- 榜单请求没有刷新模式字段，无法表达“自动读取历史”与“用户主动强刷”的差异。
- 分析结果以 30 天缓存为主，同一本书无法在不重爬章节的前提下再次触发 AI 分析。
- 趋势分析虽然已经落库，但返回结果仍偏通用，无法直接支撑词云、题材表、最近三次对比视图。
- 书籍主表按 `platform + book_url` 去重，链接失效或变更时容易新增记录，无法把“修正链接”当成同一本书的维护动作。

## 设计边界

- 本轮只改后端、SQL、测试和必要的文档，不扩展前端页面。
- 不引入新的异步任务系统，继续复用同步调用链路与现有 `crawler_task` 表。
- 不扩展新的平台抓取逻辑，也不把趋势分析改成离线批处理。

## 核心方案

### 1. 刷新决策从“纯缓存”改成“数据库快照 + 决策层”

新增一层刷新策略：

- `AUTO`：优先读数据库中的最近有效快照。
- `FORCE`：在满足冷却与次数限制时允许重新抓取。
- `REPAIR_LINK`：仅用于检测到详情或章节链接异常时，执行一次修链补抓。

榜单数据判断顺序：

1. 先看是否存在最近有效榜单快照。
2. 若存在且未超过业务刷新阈值，则直接返回数据库快照。
3. 若不存在或已过刷新阈值，再调用 Python 爬虫抓取并写入新快照。
4. 若用户请求 `FORCE`，则额外检查强刷冷却和强刷次数。

章节与详情数据判断顺序：

1. 优先读取数据库中对应书籍和章节数据。
2. 若缺失或链接疑似失效，则执行一次详情补抓更新链接。
3. 补抓成功后重试一次章节拉取。
4. 仍失败则返回清晰业务错误，不做无限重试。

Redis 继续保留，但只负责加速最近结果读取，不再决定业务是否要重抓。

### 2. 系统配置化刷新规则

将以下规则迁移到 `system_config`：

- 榜单自动刷新天数，默认 5 天。
- 详情自动刷新天数，默认 7 天。
- 章节缓存有效天数，默认 30 天。
- 榜单强刷冷却天数，默认 2 天。
- 榜单强刷窗口次数，默认 2 次。
- AI 重分析冷却小时数，默认 0 或很小值，允许短期重跑。

这样后续即使你把榜单改成 4 天或 3 天，也不需要继续改代码常量。

### 3. 榜单与强刷审计

复用 `crawler_task` 记录榜单刷新动作：

- `task_type=rank_refresh`
- `request_json` 记录平台、分类、刷新模式和原因
- `status` 记录成功/失败
- 通过最近成功任务统计强刷次数和冷却窗口

这样可以支撑：

- 当天或短期内重复点“扫榜”时，直接返回数据库快照
- 用户点击“强制抓取”时做次数与时间限制
- 后续追踪“到底什么时候真正重新抓过榜”

### 4. 书籍唯一性与修链策略

当前主表按 `platform + book_url` 去重，无法稳定表达同一本书链接更新。

本轮调整为优先按：

- `platform + platform_book_id`

兜底仍可兼容旧数据中的 `book_url`。

处理逻辑：

- 若详情补抓返回了新的 `platform_book_id` 或新链接，优先更新原有书籍记录。
- 若旧记录没有平台书籍 ID，则补回该字段。
- 只有无法命中旧记录时才创建新书。

### 5. AI 重分析与趋势分析结构化

拆文类接口新增：

- `forceReanalyze`

行为：

- `false`：仍允许复用现有分析缓存。
- `true`：跳过分析缓存，复用已抓取章节重新调用 AI，并将新结果再次入库。

趋势分析与数据分析改为返回更稳定的结构化 JSON，至少包含：

- `wordCloud`: 热门题材词条及权重
- `themeDistribution`: 题材占比分布
- `themeTable`: 题材统计表格
- `comparisonSummary`: 最近三次快照的变化总结
- `snapshotComparison`: 每次快照的主要题材变化

Java 侧负责：

- 解析模型返回 JSON
- 在模型未完全遵守结构时做兜底清洗
- 把结构化结果写入 `analysis_result.result_json`

### 6. 数据分析接口扩展

`GET /api/data/visual` 扩展返回：

- 分析类型分布
- 每日分析趋势
- 榜单分类分布
- 最近三次榜单快照
- 热门题材词云
- 热门题材统计表
- 最近三次题材对比摘要

数据来源：

- 基础计数类来自数据库聚合
- 词云与题材表优先复用最近一次 `theme` 趋势分析的结构化结果
- 若趋势分析缺失，则返回空集合而不是报错

## 接口调整

### `POST /api/crawler/rank`

新增请求字段：

- `refreshMode`：`AUTO | FORCE`
- `forceReason`：可选，记录用户强刷原因

默认：

- 不传时按 `AUTO`

### `POST /api/analysis/{type}`

新增请求字段：

- `forceReanalyze`

默认：

- 不传时为 `false`

## 数据库调整

### 修改

- `crawl_book` 增加更可靠的唯一性查询策略，优先复用 `platform_book_id`
- `crawler_task` 复用于刷新审计，不新增独立强刷表

### 配置种子

新增或补齐：

- `crawler.rank.refresh-days`
- `crawler.rank.force-cooldown-days`
- `crawler.rank.force-max-times`
- `crawler.book.refresh-days`
- `crawler.chapter.cache-days`
- `analysis.reanalyze.cooldown-hours`

## 错误处理

- 强刷次数超限：返回 `429`
- 榜单仍在有效期：返回数据库快照并附带“使用历史快照”的语义消息
- 链接失效且修链失败：返回 `400/404` 级业务错误
- AI 返回非结构化结果：保存兜底摘要，并返回空词云/题材表，而不是整条链路失败

## 测试策略

先写失败测试覆盖：

- 榜单 `AUTO` 模式命中数据库快照时不触发爬虫
- 榜单 `FORCE` 超过次数/冷却限制时被拒绝
- 同一本书 `forceReanalyze=true` 时会生成新的分析记录
- `GET /api/data/visual` 返回词云、题材表、最近三次对比结构
- 详情或章节链接失效时会触发一次修链补抓

## 实施假设

- 本轮先以“后端语义正确、接口可供前端接入”为目标，不额外实现前端按钮。
- 强刷限制先按系统配置实现，后续若前端需要更细按钮文案，再补充展示字段。
- 若模型暂时无法稳定产出完整 JSON，Java 侧会以兜底清洗保证接口结构稳定。
