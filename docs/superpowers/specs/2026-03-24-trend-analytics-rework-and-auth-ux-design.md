# 趋势分析重构与认证体验升级设计

## 目标

本轮改造解决两类核心问题：

1. 登录与注册流程缺少明确、友好的中文错误提示，注册校验过弱，前端直接暴露生硬异常信息。
2. 趋势分析功能当前偏离真实业务目标，没有围绕扫榜页中用户选中的具体榜单分类做结构化分析，也没有让结果直接驱动趋势页展示。

最终目标是把趋势分析改造成“榜单级结构化 JSON 契约驱动”的链路，同时把认证体验升级到可直接上线验收的水平，并确保桌面端与手机端都可用。

## 设计边界

- 本轮覆盖认证体验、趋势分析后端契约、趋势页前端交互、相关测试与种子数据。
- 不引入验证码注册，不扩展第三方认证。
- 不重做整套用户系统，只在现有 JWT 鉴权基础上补强注册、登录和异常语义。
- 不继续沿用“页面挂载自动发起趋势分析”的策略，趋势分析改为用户显式触发。
- 不把趋势页做成平台级泛分析，必须围绕 `platform + channelCode + boardCode` 所代表的具体榜单。

## 已确认问题

### 1. 认证体验问题

- `LoginView.vue` 只校验必填和确认密码一致，没有密码规则说明。
- 后端 `AuthService` 对注册只做用户名重复检查，没有密码强度规则。
- 登录失败统一返回英文 `"username or password is incorrect"`，无法区分“用户不存在”“密码错误”等可读场景。
- 前端直接显示后端 message 或 Axios message，容易出现奇怪数字、英文或不完整异常。

### 2. 趋势分析偏离目标

- `TrendView.vue` 进入页面就自动调用趋势分析，不符合“没点就别开始”的要求。
- 趋势分析请求只带 `platform + category`，且前端分类列表还是写死的伪分类。
- 后端 `AnalysisService.analyzeTrend(...)` 只是把最近三次榜单快照拼成原始文本，请 AI 返回自由文本，再存成普通 `theme` 分析结果。
- `DataQueryService.getVisualData(...)` 只按平台聚合，趋势页数据并不是围绕“当前榜单”构建。
- 词云、对比摘要、趋势表等数据结构松散，主要靠前端兜底猜测，无法稳定落库复用。

## 总体方案

### 1. 认证链路改造

认证链路采用“后端稳定错误语义 + 前端友好映射 + 表单规则前置提示”的方式升级。

后端侧：

- 为注册引入密码规则校验，至少包括长度下限、包含大写字母、包含小写字母、包含数字。
- 登录时区分：
  - 用户名不存在
  - 密码错误
  - 账号禁用或不可用
- 注册时区分：
  - 用户名已存在
  - 密码不符合规则
  - 默认角色缺失等系统异常
- 保持 HTTP 状态语义准确，返回给前端的 message 改成稳定中文。

前端侧：

- 在注册态展示密码规则说明和校验反馈。
- 表单提交前做基础校验，减少无效请求。
- 统一将后端异常转换为中文用户提示，不直接裸露奇怪数字、英文技术文本或不完整 trace。
- traceId 只作为辅助信息展示，不抢占主要错误提示。

### 2. 趋势分析改造成榜单级结构化 JSON 契约

趋势分析从“自由文本总结”改为“榜单级结构化分析结果”。

#### 输入上下文

趋势分析输入必须围绕用户正在关注的具体榜单：

- `platform`
- `channelCode`
- `boardCode`
- `boardName`
- 最近三次该榜单抓取快照
- 每次快照的上榜书籍名称、作者、简介、排名
- 历史已生成的该榜单趋势分析摘要，用于词云和变化判断

趋势页默认通过用户已保存的榜单偏好获取当前榜单，也允许用户在趋势页切换榜单。

#### 输出契约

AI 返回结果必须约束为稳定 JSON，并由 LangChain4j JSON 模式配合后处理解析。核心字段包含：

- `analysisType`
- `platform`
- `channelCode`
- `boardCode`
- `boardName`
- `summary`
- `shortSummary`
- `trendConclusion`
- `themeKeywords`
- `historicalWordCloud`
- `themeEvolution`
- `hotBooks`
- `snapshotComparisons`
- `insightCards`
- `riskSignals`

其中：

- `historicalWordCloud` 明确代表“该榜单历史三次趋势分析所归纳出的题材词云”。
- `themeEvolution` 表示最近三次快照的题材变化轨迹。
- `hotBooks` 用于展示代表性上榜作品及原因。
- `snapshotComparisons` 用于卡片/表格直接展示，不再让前端二次猜测。

### 3. 趋势分析存储与查询策略

趋势分析结果仍落到 `analysis_result`，但会把 `result_json` 作为主要展示数据源。

约束如下：

- `analysis_type` 继续使用 `theme`，避免额外扩表。
- `book_id` 对于榜单趋势分析改为可空或使用榜单锚点书籍，但前端查询不得依赖单书语义。
- 趋势相关接口查询时，必须按 `platform + channelCode + boardCode` 聚合筛选。
- 可视化接口不再返回平台级混合数据，而是返回“当前榜单”的结构化展示数据。
- 若当前榜单已有可复用趋势分析结果，优先复用缓存或最近结果；用户点击重新分析时才强制请求 AI。

### 4. 趋势页交互重做

趋势页行为改成：

- 进入页面只加载榜单上下文与已保存的可视化数据，不自动发起趋势分析。
- 用户点击“开始分析”才发起流式趋势分析。
- 用户切换榜单时只切换上下文，不自动重跑分析，除非用户再次点击分析或选择刷新。
- 结果默认展示约 300 字摘要，完整内容通过详情抽屉/弹层查看。
- 手机端优先使用纵向卡片、折叠区块和底部抽屉，避免长内容整页堆叠难滑。
- 详情抽屉要有明显关闭入口，并兼容手机端手势不稳定场景。

### 5. 图表与展示字段设计

图表和数据卡片必须直接消费中文字段或可稳定映射字段。

趋势页展示将重点围绕：

- 当前榜单名称与快照信息
- 最近三次快照时间
- 题材演进
- 历史词云
- 代表作品
- 风险信号
- 趋势摘要

不再展示与当前榜单无关的平台混合统计，也不再把英文 code 直接暴露到图表上。

## 数据流

1. 用户在扫榜页选择榜单，偏好被保存为 `platform + channelCode + boardCode`。
2. 进入趋势页时，先读取用户偏好和榜单目录，恢复当前榜单上下文。
3. 趋势页调用新的可视化接口，获取该榜单最近三次快照和历史结构化趋势结果。
4. 用户点击“开始分析”后，请求趋势流式分析接口。
5. 后端拉取该榜单最近三次快照，组合结构化提示词，要求模型输出 JSON。
6. AI 网关用 JSON 模式 + 解析配置解析结果，写入 `analysis_result.result_json`。
7. 流式接口优先输出文本摘要，完成后带上结构化结果；前端同步刷新趋势卡片与图表。

## 错误处理

### 认证

- 表单校验错误：前端直接中文提示。
- 用户名不存在、密码错误、用户名重复、密码不合规：后端返回明确中文 message。
- 未知系统异常：统一显示“操作失败，请稍后重试”，保留 traceId 辅助定位。

### 趋势分析

- 没有榜单偏好：提示用户先去扫榜页选择榜单。
- 当前榜单没有三次快照：提示“历史样本不足，暂时无法生成趋势分析”。
- AI 返回非 JSON 或结构不完整：后端兜底生成结构化最小结果，并记录原始文本。
- 可视化接口读取不到历史趋势数据时，前端展示空态，而不是编造图表内容。

## 测试策略

### 后端

- 认证控制器测试新增：
  - 用户不存在提示
  - 密码错误提示
  - 密码规则不合规提示
  - 用户名重复提示
- 趋势分析集成测试新增：
  - 围绕 `channelCode + boardCode` 生成趋势分析
  - 持久化结构化 `result_json`
  - SSE 接口返回结构化趋势分析完成事件
- 数据查询测试新增：
  - `/api/data/visual` 按具体榜单返回数据
  - 历史词云、题材演进、代表作品等字段可直接返回

### 前端

- 登录页测试新增：
  - 注册密码规则提示
  - 用户名不存在/密码错误/用户名重复映射
  - 不展示裸技术异常
- 趋势页测试新增：
  - 页面挂载不自动分析
  - 读取榜单偏好恢复上下文
  - 点击开始分析才发请求
  - 移动端详情抽屉可打开和关闭
  - 长结果默认只展示 300 字摘要

## 实施假设

- 当前数据库结构已具备 `rank_board`、`rank_snapshot`、`crawl_rank`、`analysis_result` 等基础表，可支撑本轮改造。
- 现有 AI 网关已经具备 JSON 输出约束能力，本轮主要是把这一能力落到趋势分析正式链路。
- 用户明确要求自主执行，因此本设计直接视为可实施基线，不再等待中间确认。
