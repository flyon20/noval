# 安全加固修复设计

## 目标

补齐当前认证与权限链路中的高风险缺口，确保登录、鉴权、续签、限流、IP 来源识别与错误响应符合生产可用的安全基线。

## 范围

- 去除默认可用的 demo 管理员回退能力。
- 收紧 JWT 默认配置，避免弱配置继续放行。
- 让登录接口进入限流与黑名单链路。
- 让 `refresh` 校验黑名单并回查用户状态与角色。
- 只在可信代理场景下信任 `X-Forwarded-For`。
- 为 401/403/429 返回真实 HTTP 状态码。

## 非目标

- 不改动业务接口的返回结构字段。
- 不引入新的认证体系或外部身份提供方。
- 不重写现有自定义权限框架为 Spring Security 全量方案。

## 设计决策

### 1. 认证配置收紧

- 新增 `demo-enabled` 配置，默认关闭 demo 登录。
- JWT 密钥不再使用仓库内固定默认值；若未配置或过短，则启动失败。
- 测试环境通过 `src/test/resources/application.yml` 提供专用测试密钥，避免污染生产默认配置。

### 2. 登录限流接入

- 复用现有 `RateLimitService`，在登录入口显式执行限流判断。
- 登录限流继续沿用现有自动黑名单策略，避免为登录单独维护一套风控逻辑。

### 3. 刷新链路收紧

- `refresh` 前先校验 token 是否已进入黑名单。
- 解析 claims 后，根据用户 ID 回查数据库中的活跃用户与角色。
- 用户不存在、被禁用或角色已变更时，不再信任旧 token 中的历史角色声明。

### 4. IP 来源解析

- 新增统一 `RequestIpResolver`。
- 默认只使用 `remoteAddr`。
- 仅当请求来自 `trusted-proxy-ips` 中的地址时，才解析 `X-Forwarded-For` 首个 IP。

### 5. HTTP 状态码修复

- 保留现有 JSON `code/message/data/traceId` 结构。
- 过滤器写回响应时，根据 `Result.code` 设置 HTTP 状态码。
- 全局异常处理改为返回对应的 `ResponseEntity<Result<?>>`，保证 400/401/403/429/500 具备真实 HTTP 语义。

## 验证策略

- 新增或更新认证与安全测试，先让以下场景失败：
  - 无 token 访问受保护接口返回 HTTP 401。
  - 权限不足返回 HTTP 403。
  - 登录高频请求触发 HTTP 429。
  - 登出后的 token 不能再用于 `refresh`。
  - 被禁用用户的旧 token 不能继续 `refresh`。
  - 未信任代理时伪造 `X-Forwarded-For` 不生效。
- 完成修复后运行 `mvn test` 全量回归。
