## 2. 项目范围与边界

### 2.1 V1 In Scope（必须实现）

1. 番茄榜单扫榜：阅读榜、新书榜及细分栏目。
2. 书籍元信息入库：书名、链接、简介、平台、抓取时间。
3. 章节抓取：按 1/3/5/10 章可选抓取与存储。
4. AI 分析：拆文、拆结构、拆情节、题材趋势。
5. 数据分析：最近三次抓取对比、词云/饼图等统计视图。
6. 用户登录鉴权：JWT、权限校验、限流、黑名单。
7. 提示词配置：按类型管理并可切换生效，支持 `modelName / temperature / maxTokens`。

补充说明：

- 当前权限模型采用轻量 RBAC（Role-Based Access Control）。
- 以 `sys_user`、`sys_role`、`sys_user_role` 为核心，通过 JWT claims 携带角色并在接口层使用 `@RequireRole` 做角色校验。
- 当前粒度以“角色级接口访问控制”为主，未扩展到资源实例级权限或 ABAC 策略引擎。

### 2.2 Out of Scope（V1 不做）

1. 起点真实抓取逻辑。
2. 引入额外遥测/埋点平台。
3. 非文本媒体（图床、音视频）管理。

## 3. 架构与降级策略

### 3.1 架构分层

1. 前端层（PWA）：Vue3 + Element Plus。
2. 后端层（Spring Boot）：鉴权、业务编排、缓存、持久化。
3. 爬虫层（FastAPI）：榜单抓取、章节抓取、清洗与反反爬策略。
4. AI 调用层（LangChain4j + OpenAI 兼容模型）：提示词渲染、模型调用与结构化输出。

### 3.2 AI 调用降级策略

- 主通道：后端 -> LangChain4j -> OpenAI 兼容模型供应商。
- 兜底通道：模型调用失败时回退本地 fallback 结果，保证链路可落库、可追踪。
- 降级要求：记录 `traceId`、失败原因、重试次数；保证请求可追踪。

## 4. API 契约（统一）

> 说明：缓存键与 TTL 是实现建议，不属于对外 API 契约。

### 4.1 统一请求约束

- 基础路径：`/api`
- 认证头：`Authorization: Bearer <token>`
- 请求体新增公共字段：`platform`
- V1 平台枚举：`fanqie`（`qidian` 预留）

请求示例：

```json
{
  "platform": "fanqie",
  "bookId": 10001,
  "chapterCount": 5
}
```

### 4.2 统一响应结构

```json
{
  "code": 200,
  "message": "success",
  "data": {},
  "timestamp": 1760000000000,
  "traceId": "f0b5c0aa2cbe4f58a7d0f8f012345678"
}
```

### 4.3 状态码

- `200` 成功
- `400` 参数错误
- `401` 未登录或 Token 过期
- `403` 无权限
- `404` 资源不存在
- `429` 请求频率超限
- `500` 服务端错误

### 4.4 提示词类型枚举（前后端统一）

`deconstruct | structure | plot | theme | comprehensive`

说明：V1 前端页面仅暴露已落地的 `deconstruct | structure | plot | theme`，详细对接见 `docs/前端接口设计-v1.md`。

## 5. 数据与缓存设计

### 5.1 表设计原则

- 表名、字段名：小写下划线。
- 主键：`BIGINT`。
- 审计字段统一：`create_time`、`update_time`、`deleted`。
- 索引命名：普通索引 `idx_`，唯一索引 `uk_`。

### 5.2 密钥与敏感信息

- 禁止明文存储 `dify_api_key`。
- 改为 `dify_api_key_ref`（密钥引用 ID）或运行时环境变量注入。
- 日志中禁止输出 Token、密钥、手机号完整值。

补充：

- 前端永不回传或展示真实模型 key，只展示“已配置”状态或掩码信息。
- 后台通过 `system_config` / `ai.model-registry.json` 保存的密钥必须以服务端加密密文落库，运行时由后端解密后使用。
- 生产环境优先使用 `CONFIG_SECRET_MASTER_KEY` 作为配置密钥主密钥；未显式提供时可回退到 JWT 密钥，但不建议长期共用。
- 本地配置文件可以继续使用环境变量引用；若必须在后台录入真实 key，也只能由后端持有，前端不得缓存明文。

### 5.3 本轮数据库影响

- 不新增表，不改表结构。
- 继续使用 `prompt_config` 的 `model_name`、`temperature`、`max_tokens` 字段。
- 在 `system_config` 中新增 AI 配置种子项，如 `ai.provider.type`、`ai.openai-compatible.default-model`。

### 5.4 缓存建议（非 API 契约）

- 榜单：3 天
- 书籍详情：7 天
- 章节与分析结果：30 天
- 趋势统计：1 天
- 限流计数：按分钟/小时窗口

## 6. 阿里巴巴规范落地清单

### 6.1 Java 规范

1. 包名使用全小写反域名，如 `com.novelanalyzer.modules.analysis`。
2. 分层命名明确：`DTO`（入参）、`VO`（出参）、`DO/Entity`（持久化）。
3. 统一异常体系：业务异常 + 全局异常处理 + 错误码分层。
4. 禁止 `System.out.println`，统一日志框架输出。
5. 日志必须参数化占位，避免字符串拼接。

### 6.2 SQL 规范

1. 必要索引覆盖高频查询条件。
2. 禁止 `select *`，字段按需选择。
3. 软删字段统一为 `deleted`（`0/1`）。
4. 变更脚本必须可回滚。

### 6.3 日志与安全规范

1. 敏感字段脱敏（手机号、邮箱、Token、密钥）。
2. 登录失败、权限拒绝、限流触发需记录审计日志。
3. 核心接口（AI 调用、提示词配置）必须鉴权 + 限流。
4. 黑名单策略文档化：触发条件、封禁时长、解封流程。

## 7. 部署与环境策略

- 本地开发：允许先采用非 Docker 方式快速开发。
- 联调/测试/生产：统一 Docker Compose。
- Nginx 反向代理在联调与生产启用。

## 8. Git 协作规范（简化 GitFlow）

### 8.1 分支角色

- `main`：稳定可发布
- `dev`：日常集成
- `feature/*`：功能开发
- `release/*`、`hotfix/*`：按需启用

### 8.2 推荐命名

- `feature/<功能描述>-<日期>`
- `release/v<版本号>`
- `hotfix/<问题描述>-<日期>`

### 8.3 提交规范

- 推荐前缀：`feat`、`fix`、`docs`、`refactor`、`test`、`chore`
- 示例：`docs: 完成项目总设计v2与分步开发计划`

## 9. V1 验收基线

1. 用户登录后可完成番茄榜单抓取并查看结果。
2. 可按章节数量抓取正文并触发三类分析。
3. 分析结果可查询、可回看、可做最近三次对比。
4. 提示词配置可按类型切换并生效。
5. 核心接口具备鉴权、限流、审计日志。
6. 文档与实现一致，无范围冲突与技术栈冲突。
